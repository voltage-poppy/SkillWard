# 对比参考

我们在同一组 Skills 数据集上运行了两个公开工具作为参考基线。这些工具在代码级与策略级检查上具有参考价值，但都不包含隔离的运行时验证。

---

## 产品A（静态 + LLM，无沙箱）

产品A 是一个成熟的 Skill 安全扫描工具，采用静态分析 + LLM 研判的双层架构。

| 结论 | 占比 |
|------|------|
| **SAFE** | 61.3% |
| **UNSAFE** | 38.7% |

**优势：**
- 静态分析覆盖面较广，包含命令注入、数据外传、提示词注入等多类检测规则
- LLM 研判层能理解 Skill 语义意图，不完全依赖模式匹配
- 开源成熟，社区活跃

**局限：**

- **误报：** 纯静态规则无法理解代码上下文和意图 — 例如 `node_modules` 中的标准库代码（如 `es-abstract/function-bind`）触发命令注入规则；中日韩文本被误识别为同形字攻击；安全教学文档被当作真实的提示词注入；正常的本地文件写入被标记为数据外传
- **漏报：** 仅在 Skill 被执行的过程中才暴露的威胁完全无法被检测到 — 如通过 `curl` 外泄凭证、`crontab` 持久化后门、`postinstall` 供应链攻击等
- **噪音：** 总发现数中约 88.6% 为 INFO 级别噪音，需要人工筛选

---

## 产品B（静态匹配 + 云端 API，无沙箱）

产品B 是一款扫描工具，提供 CLI 和 Web UI 两种使用方式。本地端负责收集 Skill 内容，发送到云端 API 进行分析。

| 结论 | 占比 |
|------|------|
| **SAFE** | 45% |
| **UNSAFE** | 55% |

**优势：**
- 支持自动发现本机上的多种 AI Agent（Claude Code、Cursor、VS Code、Windsurf、Gemini CLI 等）
- 能正确识别部分真实威胁（硬编码 API 密钥、链上交易提交、pipe-to-shell 安装等）
- Web UI 免费使用，降低了上手门槛
- 与多个第三方平台有集成

**局限：**

- **误报：** 基于关键词匹配而非行为理解，导致误报率偏高。例如：
  - 提到"Binance"就标记为金融风险，不区分只读查询和实际交易执行
  - 提到"web_search"就标记为第三方内容暴露，不区分静态文档模板和真实网络请求
  - 纯文本决策框架（零代码、零网络调用）因提到"external perspective"被标记为威胁
- **分级粗糙：** 绝大多数 issue 统一标记为 HIGH，一个只读行情查询工具和一个实盘交易 bot 获得相同的严重性评级，用户无法据此做出有效的安全决策
- **漏报：** 缺乏运行时验证，隐藏在动态构造命令、运行时下载依赖或伪装内容中的威胁无法被发现
- **依赖云端：** 核心分析逻辑在服务端，离线环境无法使用

---

## 总结对比

| 维度 | SkillWard | 产品A | 产品B |
|------|-----------|-------|-------|
| **检测架构** | 静态 + LLM + Docker 沙箱 | 静态 + LLM | 静态匹配 + 云端 API |
| **运行时沙箱** | Docker 隔离执行 + 蜜罐诱饵 | 无 | 无 |
| **误报控制** | 沙箱验证过滤灰区，降低误报 | 较多误报 | 关键词驱动，误报率偏高 |
| **严重性分级** | SAFE / ALERT / DANGER 三级 | SAFE / UNSAFE 二级 | 几乎全部标记为 HIGH |
| **检测覆盖** | 已知模式 + 语义意图 + 运行时行为 | 已知模式 + 语义意图 | 关键词 + 云端规则 |
| **Agent 自动发现** | 否（扫描指定目录） | 否 | 支持（自动发现本机 Agent 配置） |

> **核心差异：** 产品A 和 产品B 都在代码级检查上有各自的优势，但两者都缺少运行时验证。SkillWard 在静态和 LLM 的基础上引入 Docker 沙箱实测，把无法通过代码审查确认的灰区告警转化为高置信度的运行时证据。
